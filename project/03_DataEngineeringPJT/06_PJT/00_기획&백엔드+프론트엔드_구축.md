# 프로젝트 기획 및 백엔드&프론트엔드 프로젝트 구축 (Vue와 Django)
## 목표
팀원과 프로젝트 주제를 선정하고, 해당 서비스에서 보여야 할 내용에 대한 구체적인 계획을 세웁니다.

### 최종 관통 프로젝트 목표
Django(백엔드)와 Vue.js(프론트엔드)를 활용하여 데이터를 관리, 표시하고, 사용자 계정과 활동(예: 뉴스 기사를 대상으로 한다면 기사 좋아요, 읽은 기록 체크 등 서비스와 관련된 정보)을 시각화하는 기능을 기획 및 설계하는 것입니다.

- 단순 웹 서비스(ex. 게시판)는 CRUD만 구성하면 되지만, 이를 넘어서 CRUD + 데이터 처리, 가공, 저장하는 요소까지 고려해서 기획해야 함

- 외부에서 수집하는 데이터 외에도, 사용자 로그 등 백엔드에서 수집되는 데이터도 함께 고려할 것

### 세부 목표
해당 프로젝트에서는 Kafka(카프카)와 Flink(플링크)가 내부적으로 돌아가는 환경으로 가정하며, 이미 지속적으로 수집되면서 들어오는 데이터를 Vue 프론트엔드로 출력하고 사용자 로그인/회원가입, 좋아요 기능 등 기본적인 서비스 요소까지 구현합니다.

- Django REST Framework로 API를 설계하고 Vue.js에서 웹 화면에 표시할 수 있다.
- 이후에 구현한 Kafka + Flink 파이프라인에서 실시간으로 처리되는 서비스 DB를 연동할 수 있다.
- 사용자 활동(읽은 기사, 좋아요 등)을 저장 후 대시보드에서 시각화 할 수 있다.
- 추가로 API 설계 문서(API Sheet)를 작성하여 팀원 간 협업과 유지보수 용이성 확보할 수 있다.

- 정리
  1. 웹 서비스 기획
  2. 웹 서비스 DB 설계
  3. 수집할 데이터 가능 여부
      - (준)실시간 형태로 수집 가능한 데이터인지 확인
  4. 일반 데이터 조회(RDB) + 검색엔진(검색용 텍스트) 부착
  5. "챗봇기능" 적용할만한 컨텐츠 있는지
      - 챗봇이 AI 접목 수월함. 다른거 해도 상관X
      - AI 적용한 기능이 하나 이상 있어야 함
  

## 준비사항
### 사용 데이터
- 카프카 & 플링크 파이프라인을 통해 수집될 데이터 가정 (PostgreSQL 등 DB에 저장)  
- 크롤링, 지속적 업데이트 되는 다양한 경로를 통해 수집된 정보  

### 개발언어 / 프로그램
- `Django REST Framework`: 백엔드 API 구성  
- `Vue.js`: 프론트엔드 페이지 구성  
-` Kafka & Flink`: 이후 구축할 스트리밍 파이프라인 (데이터 관리)  
- `PostgreSQL`: 서비스를 위해 수집된 데이터 및 사용자 정보 저장
- API 문서화 툴(`Swagger`, `Postman`, `Notion` 등): API Sheet 작성 시 활용 가능
  - endpoint 기반 설계 필요?
  - GET, POST -> `api/news/{news_id}`


## 구현 방법
### 1) 프로젝트 기획 및 설계
- 서비스 주제 결정: 실시간 데이터 수집 및 분석을 통한 서비스 개발 (예: 날씨 정보, 금융 데이터, 교통 상황 등)  
- OLTP 데이터베이스 설계 (서비스 테이블, 사용자 테이블 등) 모델링 반영
- 디테일, 좋아요 처리, 회원가입/로그인 등 주요 엔드포인트 구현  

### 2) 백엔드(Django) 초기 설정 및 구축
- Django 프로젝트 생성, Django REST Framework 설치 및 API 구조 설계  
- 데이터베이스(서비스테이블 설계, 사용자 테이블 설계 등) 모델링 반영
  - ORM 모델링을 통해 DB 구조 설계 진행과 동시에 백엔드에서 endpoint 어떻게 가져갈 것인지 
- 데이터 목록, 디테일, 좋아요, 회원가입/로그인 등 주요 엔드포인트 구현  

### 3) 프론트엔드(Vue.js) 구성
- Vue CLI로 프로젝트 생성
- Vue 기본 구조, component, router, view
- Vue Router 설정(페이지 간 이동 담당) → 로그인 페이지, 메인 페이지, 서비스 목록 및 상세 페이지, 대시보드 페이지
  - axios(HTTP 통신 라이브러리) 통해서 django API 호출
    - `axios.get('api/data')`
    - 받은 데이터를 Vue component에서 렌더링

### 4) DB 구축 및 서비스 데이터 연동
- 앞으로의 수집할 데이터 환경을 생각하며 카프카&플링크 환경을 그대로 활용  
- Flink Consumer를 통해 DB (PostgreSQL)에 저장될 데이터(예: 뉴스 기사)를 Django에서 조회하여 Vue에 전달  
- 지속적으로 실시간 데이터를 수집할 수 있는 구조를 통해 새로 들어오는 데이터도 지속적으로 DB에 누적 → 실시간/최신 서비스 제공  

### 5) 로그인/회원 관리
- Django에서 JWT 또는 Session 기반 인증 구성  
- Vue 페이지에서 회원가입/로그인 UI 및 요청 로직 구현  
- 로그인 후, 좋아요나 사용자별로 저장될 수 있는 요소가 서비스화 될 수 있도록 백엔드 연동  

### 6) 대시보드(사용자 활동 시각화)
- 사용자별 활동: 좋아요한 기사 목록, 최근 읽은 기사 목록, 혹은 통계 그래프
- Vue에서 Chart.js, ECharts, Plotly 등 라이브러리를 활용한 시각화  
- Django에서 REST API로 통계 데이터 제공  


## 관통 프로젝트 가이드 (예시)
### PJT 내용
- 데이터 수집
  - Kafka → Flink → PostgreSQL 파이프라인으로 뉴스 데이터가 DB에 쌓임

- 백엔드 Django 구성
  - DB와 연동, REST API (뉴스 목록, 상세, 좋아요, 사용자 관리 등) 구현

- 프론트엔드 Vue.js 구성
  - Axios로 백엔드 API 호출하여 뉴스 및 사용자 정보 렌더링  
  - 라우팅/로그인/로그아웃 등 구현

- 활동 기록 및 대시보드
  - 사용자 활동(좋아요/읽기 기록) DB 저장 → Vue 차트 라이브러리로 시각화


### 요구사항
- 기본 기능
  - DB와 서비스 데이터 연동, Vue 화면에 띄워주기  
    - Django에서 데이터 목록/상세 API 구현  
    - Vue에서 데이터 목록 페이지, 상세 보기 페이지 구현  
    - Kafka+Flink로부터 계속 쌓이는 DB 데이터가 반영되도록 고려해야 함

- 요청 조건
  - Django REST Framework 기반 API  
  - Vue CLI/Router로 페이지 분리  
  - 뉴스 추천 서비스를 예로 들어 기사 제목, 요약, 발행일 등 주요 정보 노출  

- 결과
  - DB에 저장된 데이터 목록 설계 및 테이블 구현  
  - Vue 화면에서 확인 가능한 더미 데이터 테스트  
  - 주기적으로 수집되는 최신 데이터 자동 반영을 고려한 설계  

### 추가기능
- 로그인/회원관리 기능 및 API Sheet 작성
  - 로그인/회원관리 기능 및 API Sheet 작성  
  - Django Authentication(JWT 또는 세션)으로 회원관리  
  - Vue에서 로그인 페이지 구현  
  - 추가로 API Sheet 작성 (엔드포인트 명세, Request/Response 파라미터 예시 포함)

- 요청 조건
  - 회원가입, 로그인, 로그아웃, 비밀번호 해싱 등 기본 보안 고려  
  - API Sheet(Swagger, Postman 문서 등)로 각 엔드포인트 정리  

- 결과
  - 로그인 상태에 따라 페이지별/기능별 접근 권한을 다르게 설정된 서비스
  - API Sheet를 통해 전체 API 스펙 정리  

### 심화기능
- 대시보드에서 나의 활동 시각화
  - “좋아요”한 콘텐츠, “확인한” 콘텐츠 등의 기록을 DB에 저장  
  - Vue 대시보드에서 사용자별 활동을 그래프로 표현  

- 요청 조건
  - 좋아요, 읽기 시점 등을 테이블에 저장  
  - Vue 차트 라이브러리 (Chart.js, ECharts 등) 사용  
  - 접속한 사용자의 활동 통계 제공 (가장 많이 본 카테고리, 좋아요 횟수 등)

- 결과
  - 사용자별 웹 대시보드에서 활동 이력 조회 및 그래프 시각화 가능

### 데이터 기반 웹 서비스 기획
- 서비스가 어떻게 구조적으로 연결되는지 이해 -> 구현
- OLTP, OLAP 따로 구현하여 서비스에 반영
  - OLTP: 사용자 활동, 좋아요, 댓글 등 실시간 처리
  - OLAP: 통계, 대시보드 등 분석용 데이터
    - 배치 형태로 airflow 등으로 주기적 업데이트 -> 장기 저장소에 저장하는 형태 고려
- PostgreSQL 등 RDBMS 활용
  - Postgres 권장
    - vecotor 저장도 가능 -> AI 기반 서비스 추가 가능
- 검색 기능 필수
  - Elasticsearch 기반 검색 요소 구현
- 최종
  - 데이터 수집 -> 가공 -> 저장 -> 웹에서 분석 및 시각화
    - kafka + flink으로 수집하여 실시간(준실시간) 데이터 구하기(ex. 웹 크롤링 등)
- 역할 구분
  - 벡엔드
    - 데이터 처리, 저장
  - 프론트
    - 사용자 화면
  - 사용자 요청이 들어왔을 때 어떤 API 형태로 백으로 들어감. 이때 파이프라인 처리될건지, 바로 프론트로 넘길건지 등에 대한 APi 설계
  - 백, 프론트, 데이터로 구성된 시스템을 유기적으로 연결하는 것이 최종 목표
